# 🎯 Our Mission

The Cybersecurity Handbook is an open-source educational project designed to bridge the gap between:

- Academic learning
- Practical cybersecurity
- Enterprise security
- Career preparation

Every contribution should improve one or more of these goals.

- 📘 New chapters

- 💼 Career roadmaps

# 🚫 Before You Start

Please:

- Read the README.
- Read `docs/HANDBOOK_ARCHITECTURE.md`.
- Read `docs/STYLE_GUIDE.md`.
- Search existing Issues before creating a new one.

---

# 🌿 Branching Strategy

Never commit directly to `main`.

Use feature branches.

Examples:

```
feature/module-01-authentication
feature/lab-threat-modeling
feature/glossary-api-security
fix/diagram-links
docs/readme-improvements
```

---

# 📝 Commit Message Convention

We follow **Conventional Commits**.

Examples:

```
feat(module-01): add authentication chapter

docs(glossary): add OAuth terminology

fix(diagrams): correct CIA triad image

refactor(template): simplify chapter template
```

---

# 📚 Documentation Standards

All documentation must follow:

- `docs/STYLE_GUIDE.md`
- Markdown formatting
- Clear headings
- Practical examples
- Enterprise relevance
- Correct grammar

---

# 🖼️ Diagram Standards

Preferred formats:

- SVG
- PNG

Naming:

```
authentication-flow.svg

secure-sdlc.png

jwt-lifecycle.svg
```

Requirements:

- High contrast
- Readable fonts
- Consistent colors
- Accessible design

---

# 🧪 Lab Standards

Each lab must include:

- Objective
- Scenario
- Prerequisites
- Environment
- Instructions
- Expected Results
- Cleanup
- Learning Outcomes

Labs must only involve **authorized and ethical** security testing.

---

# 🎤 Interview Content

Interview questions should be categorized as:

- Beginner
- Intermediate
- Advanced
- Scenario-Based
- Enterprise

Include expected discussion points where appropriate.

---

# 🔍 Pull Request Process

Before opening a PR:

- [ ] Content reviewed
- [ ] Grammar checked
- [ ] Internal links verified
- [ ] References validated
- [ ] Diagrams included (if applicable)
- [ ] Follows Style Guide

---

# ✅ Review Checklist

Reviewers should verify:

- Technical accuracy
- Editorial quality
- Formatting consistency
- Practical relevance
- Enterprise applicability
- Accessibility

---

# 🐞 Reporting Issues

When opening an Issue, include:

- Description
- Steps to reproduce (if applicable)
- Expected behavior
- Screenshots (if applicable)
- Suggested improvements

---

# 🔒 Security

Please do **not** disclose real vulnerabilities publicly.

Refer to `SECURITY.md` for responsible disclosure procedures.
